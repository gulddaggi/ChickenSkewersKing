# ChickenSkewersKing AI 작업 규칙

## 목적
이 저장소는 Unity 게임 프로젝트다.
에이전트는 안전하고 검토 가능한 **코드 중심 변경**만 수행한다.

## 현재 개발 방향
이 프로젝트는 게임잼 프로토타입에서 출발해,
더 완성도 있는 게임으로 확장하는 단계에 있다.

현재 우선순위는 다음과 같다.

1. 프로젝트 구조 안정화
2. 게임 흐름 정리
3. 테스트 가능한 코드 추가
4. 위험한 씬/에셋 수정 최소화

## 수정 허용 범위
다음 경로만 기본적으로 수정 가능하다.

- Assets/Scripts/**
- Assets/Tests/**
- Docs/**
- tools/**

## 기본 수정 금지 범위
다음 경로는 기본적으로 수정하지 않는다.

- Assets/**/*.unity
- Assets/**/*.prefab
- Assets/**/*.mat
- Assets/**/*.anim
- Assets/**/*.controller
- Assets/**/*.asset
- ProjectSettings/**
- UserSettings/**
- Library/**
- Temp/**
- Logs/**

## 사전 승인 필요 항목
다음 항목은 수정 전에 반드시 설명하거나 승인을 요청한다.

- Packages/manifest.json
- Packages/packages-lock.json
- Assets/Scenes/** 아래 파일
- Assets/Resources/** 아래 파일
- 여러 게임플레이 시스템에서 공통으로 사용하는 public API

## 작업 원칙
- diff는 작고 국소적으로 유지한다.
- 폴더 구조를 대규모로 바꾸지 않는다.
- 파일 이동/이름 변경은 최소화한다.
- scene, prefab은 명시적 요청이 없으면 수정하지 않는다.
- 가능하면 기존 코드 전체를 갈아엎기보다, 새 스크립트를 추가하거나 작은 단위로 개선한다.
- 게임플레이 로직은 가능하면 MonoBehaviour 의존 코드와 순수 C# 로직을 분리한다.
- 구조를 과하게 복잡하게 만들지 않는다.
- enum, 상태 전이, 데이터 클래스, 단순 매니저 구조를 우선한다.
- 불확실하면 더 작은 수정안을 선택한다.

## 테스트 원칙
의미 있는 변경 후에는 다음을 지킨다.

1. 컴파일 에러를 만들지 않는다.
2. EditMode test를 실행한다.
3. 어떤 파일을 왜 바꿨는지 짧게 요약한다.
4. 확인하지 못한 부분이나 위험 요소를 명확히 적는다.

## 완료 기준
작업은 아래 조건을 만족해야 완료로 본다.

- 금지 경로를 수정하지 않았다.
- 변경 범위가 과도하게 크지 않다.
- EditMode test를 실행했거나, 실행하지 못한 이유를 분명히 적었다.
- 변경 요약이 있다.

## 권장 작업 순서
1. 관련 스크립트를 먼저 읽는다.
2. 가장 작은 안전한 수정안을 제안한다.
3. 수정한다.
4. EditMode test를 실행한다.
5. 결과와 위험 요소를 요약한다.
