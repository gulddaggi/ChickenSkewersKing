# ChickenSkewersKing AI 작업 규칙

## 목적
이 저장소는 Unity 게임 프로젝트다.
에이전트는 안전하고 검토 가능한 **코드 중심 변경**만 수행한다.

## 현재 개발 방향
이 프로젝트는 게임잼 프로토타입에서 출발해,
**출시용 캠페인형 닭꼬치 푸드트럭 타이쿤**으로 리빌드하는 단계에 있다.

현재 우선순위는 다음과 같다.

1. 기존 코어 조리 루프 재사용 범위 확정
2. 프로젝트 구조 안정화 및 역할 분리
3. Day / Chapter / Campaign 흐름 도입
4. 주문 / 경제 / 평점 / 성장 시스템 데이터화
5. 테스트 가능한 코드 추가
6. 위험한 씬 / 에셋 수정 최소화

## 현재 기준으로 우선 보존할 코어
다음 영역은 현재 프로젝트의 핵심 손맛을 구성하므로,
가능하면 **보존 또는 점진 리팩토링**을 우선한다.

- 꼬치 제작 흐름
- 도마 배치 흐름
- 버너 이동 흐름
- 굽기 액션(QTE) 흐름
- 탄 상태 / 완성 상태 전환

현재 레포 기준으로는 대체로 아래 스크립트들이 이에 해당한다.

- `Assets/Scripts/Skewer.cs`
- `Assets/Scripts/Board.cs`
- `Assets/Scripts/Burner.cs`
- `Assets/Scripts/ButtonAction.cs`
- `Assets/Scripts/ButtonActionTimer.cs`
- `Assets/Scripts/FoodMaker.cs`
- `Assets/Scripts/SpeakToSkewer.cs`

## 현재 기준으로 분리 / 치환 허용 범위
다음 영역은 게임잼 구조의 흔적이 강하므로,
출시용 구조에 맞게 **분리, 축소, 치환**이 가능하다.

- `Assets/Scripts/GameManager.cs`
- `Assets/Scripts/OrderCreator.cs`
- 단판 점수 중심 흐름
- 하드코딩 레시피 / 주문 비교 로직
- 직접 연결식 UI 활성화 로직

## 데이터 자산 관련 원칙
출시 버전으로 확장하기 위해 ScriptableObject 또는 데이터 자산이 필요할 수 있다.
다만 아래 원칙을 따른다.

- 기존 에셋의 대규모 수정은 피한다.
- 새 데이터 자산이 꼭 필요할 때만 생성한다.
- 데이터 자산 생성 전에는 어떤 구조가 필요한지 먼저 설명한다.
- 단순 데이터 구조 검증 단계에서는 우선 순수 C# 데이터 클래스를 고려한다.

## 사전 승인 필요 항목
다음 항목은 수정 전에 반드시 설명하거나 승인을 요청한다.

- Packages/manifest.json
- Packages/packages-lock.json
- Assets/Scenes/** 아래 파일
- Assets/Resources/** 아래 파일
- Assets/**/*.asset 생성 또는 수정
- 여러 게임플레이 시스템에서 공통으로 사용하는 public API의 파괴적 변경
- 파일 이동 / 이름 변경 / 폴더 대규모 개편

## 작업 원칙
- diff는 작고 국소적으로 유지한다.
- 폴더 구조를 한 번에 대규모로 바꾸지 않는다.
- 파일 이동 / 이름 변경은 최소화한다.
- scene, prefab은 명시적 요청이 없으면 수정하지 않는다.
- 가능하면 기존 코드 전체를 갈아엎기보다, 새 스크립트를 추가하거나 작은 단위로 개선한다.
- 게임플레이 로직은 가능하면 MonoBehaviour 의존 코드와 순수 C# 로직을 분리한다.
- 구조를 과하게 복잡하게 만들지 않는다.
- enum, 상태 전이, 데이터 클래스, 단순 매니저 구조를 우선한다.
- 불확실하면 더 작은 수정안을 선택한다.
- 기존 코어 손맛을 훼손하는 변경보다, 상위 구조를 분리하는 방향을 우선한다.
- UI 레이아웃 / 배치는 placeholder 수준으로만 다루고, 미세 조정은 직접 작업 전제로 둔다.

## 구현 방향 원칙
에이전트는 아래 방향을 우선적으로 따른다.

1. 점수형 단판 구조보다 캠페인형 Day 루프를 우선한다.
2. 하드코딩 배열 비교보다 데이터 기반 레시피 / 주문 구조를 우선한다.
3. 싱글톤 집중 구조보다 역할 분리된 매니저 / 서비스 구조를 우선한다.
4. UI 직접 연결보다 상태 기반 흐름을 우선한다.
5. 레거시 로직은 재사용 가치가 있는 코어만 살리고, 상위 규칙 구조는 재설계 가능하다.

## 테스트 원칙
의미 있는 변경 후에는 다음을 지킨다.

1. 컴파일 에러를 만들지 않는다.
2. EditMode test를 실행한다.
3. 어떤 파일을 왜 바꿨는지 짧게 요약한다.
4. 확인하지 못한 부분이나 위험 요소를 명확히 적는다.

## 완료 기준
작업은 아래 조건을 만족해야 완료로 본다.

- 금지 경로를 수정하지 않았다.
- 변경 범위가 과도하게 크지 않다.
- EditMode test를 실행했거나, 실행하지 못한 이유를 분명히 적었다.
- 변경 요약이 있다.
- 기존 코어 루프를 훼손했다면 그 이유와 대체 구조를 설명했다.

## 권장 작업 순서
1. 관련 스크립트를 먼저 읽는다.
2. 현재 작업이 코어 루프 보존인지, 상위 구조 분리인지 먼저 판단한다.
3. 가장 작은 안전한 수정안을 제안한다.
4. 수정한다.
5. EditMode test를 실행한다.
6. 결과와 위험 요소를 요약한다.
